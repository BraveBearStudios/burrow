# Burrow v1.0 — UAT Checklist

The deferred / manual user-acceptance items for the v1.0 MVP, gathered from every phase's
`VERIFICATION.md` + `HUMAN-UAT.md`, the milestone audit, and the UI review. The **CI-provable
contracts already passed** (api 173, ui 97, Playwright e2e green, reuse 275/275) — this list is
only the things CI cannot prove on its own.

Each item is tagged with **how** to verify it. Two of the four lanes need **no homelab** — they run
the app locally over the hermetic Fake provider:

| Tag | Lane | Needs |
|-----|------|-------|
| 🖥️ `[CC-PREVIEW]` | **Claude Code thick-client preview** — open the running UI in the desktop app's preview pane and eyeball it. Best for *visual / rendering* items. | Local Fake stack |
| 🌐 `[CHROME-AUTO]` | **Claude in Chrome** — drive the running UI to execute and assert *clickable flows* over the Fake. Best for *journeys*. | Local Fake stack |
| 🔧 `[CI]` | **First GitHub Actions run** — the runner is the authority (Docker/Trivy/shellcheck unavailable on the Windows dev host). | A PR/push |
| 📦 `[CD]` | **Real release** — a `v*` tag → GHCR publish + signature/attestation verify. | GHCR + OIDC |
| 🏠 `[HOMELAB]` | **Dev-homelab smoke** — real Proxmox + the golden template. CI never touches real infra by design. | Real Proxmox |

> Items marked **(v1.1 — currently fails)** are advisory polish not yet implemented; verifying them
> now will fail. They are tracked in `PROJECT.md` Active + `milestones/v1.0-MILESTONE-AUDIT.md` tech_debt.

## Lane A — Local UI UAT over the Fake provider (NO homelab)

Everything here runs against `BURROW_COMPUTE=fake` + a stub ttyd — the same hermetic stack the
Playwright e2e uses. Stand it up either way:

**One-shot (recommended for browser UAT):**
```
docker compose -f docker-compose.e2e.yml up
# → UI at http://localhost:4173 (nginx), API (Fake) at :8000, stub ttyd at :7681
```

**Or three local processes** (from `ui/playwright.config.ts`):
```
# 1) stub ttyd
cd api && python -m tests.e2e.stub_ttyd_server --host 127.0.0.1 --port 7681
# 2) control plane over the Fake provider
cd api && BURROW_COMPUTE=fake BURROW_DB=sqlite BURROW_E2E_TTYD_HOST=127.0.0.1 \
         uv run uvicorn main:app --host 127.0.0.1 --port 8000
# 3) UI dev server
cd ui && npm run dev      # http://localhost:5173  (or: npm run build && npm run preview → :4173)
```

Point the **Claude Code preview** at that URL for the 🖥️ items; point **Claude in Chrome** at it for
the 🌐 items.

### A.1 — Visual checks 🖥️ `[CC-PREVIEW]` (eyeball the rendering)

- [ ] **V1 · Activity drawer rendering** (UI-06) — open a workspace's activity drawer: right-side
  slide-in; the four states (loading shimmer / empty / list / poll-error strip); newest-first order;
  color-coded badges keyed to the real namespaced event types (`workspace.*`, `terminal.*`,
  `boot.error`, `reaper.*`); `boot.error` rows emphasized (2px `--err` bar + tint + red mono reason);
  redacted `data` rendered as-is. _Source: 04-UI-SPEC / 04-UI-REVIEW (22/24)._
- [ ] **V2 · Drawer responsive width** (UI-06) — at 375px the drawer should be a **full-width sheet**;
  640–1023px → `min(360px,80vw)`; ≥1024px → 360px. **(v1.1 — currently fails:** `ActivityDrawer.tsx`
  hardcodes a single `min(360px,100vw)` constant → ~15px scrim sliver at phone width.) _P4-UI1._
- [ ] **V3 · `--accent-line` focus ring** (UI-06) — keyboard-focus the trigger, the close `×`, and the
  scroll region: each should show the 2px Burrow accent focus-visible ring. **(v1.1 — currently
  fails:** no `:focus-visible` rule in `index.css`; the `<aside>` sets `outline:none` → UA-default
  focus only.) _P4-UI2._
- [ ] **V4 · Custom 3px scrollbar** (UI-06) — scroll a long event list: the thin neutral scrollbar per
  spec. **(v1.1 — currently fails:** no `::-webkit-scrollbar` / `scrollbar-*` rules → UA default.) _P4-UI3._
- [ ] **V5 · App shell + four themes** (UI-01..04) — the sidebar workspace list with live status, the
  Navbar per-node capacity chips, the StatusBar counts/uptime, the New-Workspace modal, and all four
  themes render to the 02-UI-SPEC. _Phase 2._

### A.2 — Flow checks 🌐 `[CHROME-AUTO]` (automate the journeys over the Fake)

- [ ] **A1 · Create workspace** (WS-01, UI-01/UI-03) — fill the New-Workspace modal → submit → the
  workspace appears in the sidebar and walks creating→running (cosmetic boot progress). _Phase 1/2._
- [ ] **A2 · Open activity drawer** (UI-06) — open the drawer on a workspace; assert events render
  newest-first with the correct badges and `boot.error` emphasis. _(Pairs with V1.)_
- [ ] **A3 · Terminate actually destroys** (WS-08, UI-05) — click terminate (×) → confirm → assert a
  `DELETE /api/v1/workspaces/{id}` fires **and** the workspace is gone server-side on the next list
  fetch (not just removed from the mosaic). _Audit-fixed regression (commits 25e54bb / bdd72d8); the
  guard test must fail if `onTerminate` reverts to a client-only `closePanel`. **REGRESS-WS08.**_
- [ ] **A4 · Tile / split / persist** (UI-02, UI-05) — open multiple panels, split H/V, drag, resize;
  refresh the page → the Mosaic layout reconciles (gone leaves dropped, active retargeted). _Phase 2._
- [ ] **A5 · Detach → reconnect** (UI-05, TERM-06) — close the tab, reopen, re-open the same
  workspace's terminal → it reattaches (over the stub ttyd). _Phase 2._
- [ ] **A6 · Terminal echo + reconnect overlay** (TERM-05/06/07) — type into a panel → input echoes over
  the stub ttyd frames; drop the connection → the reconnecting overlay shows and recovers. _Phase 2._
- [ ] **A7 · Stop / start — DECIDE** (WS-06, WS-07) — the `/stop` + `/start` endpoints and
  `useStopWorkspace`/`useStartWorkspace` hooks exist but **no UI invokes them**. Either (a) record that
  v1 stop/start is intentionally reaper/backend-only, or (b) add a per-workspace stop/start control and
  drive it over the Fake (stop → `stopped`, start → `running`). _WS0607-UI / PROJECT Active #1._
- [ ] **A8 · Responsive viewport assertion** — once V2 lands, add a Playwright/Chrome assertion at a
  375px viewport that the drawer width == the viewport width (and the tablet band), so criterion 10 is
  provable and can't silently regress. _(v1.1, depends on V2.)_ _P4-UI4._

## Lane C — Real release 📦 `[CD]` (GHCR + OIDC)

- [ ] **D1 · GHCR publish + signature + attestation** (CICD-05) — push a real `v*` tag → `release.yml`
  pushes both images to GHCR **by digest**, then for each digest:
  - `cosign verify --certificate-oidc-issuer https://token.actions.githubusercontent.com --certificate-identity-regexp ... <digest>` passes (keyless / OIDC);

  _P4-HV5. release.yml is structurally correct (exactly `contents:read + packages:write + id-token:write + attestations:write`, SHA-pinned actions)._

## Lane D — Dev-homelab smoke 🏠 `[HOMELAB]` (real Proxmox + golden template)

The big deferred block — real LXC operations that the Fake provider cannot exercise. **H9 is the
headline acceptance gate** (PRIMING.md STEP 4); the rest are its load-bearing parts plus the Phase-3/4
real-infra halves.

**Host + template prep**
- [ ] **H1 · Host-prime kit (idempotent)** (SETUP-01..05) — run `cc-worker-config/lxc/host-prime/{00..40}` +
  `lib/common.sh` per PRIMING.md on a bare Proxmox host, then re-run to confirm idempotency; operator
  reaches `/health` `compute:ok` and a hand `--full` clone works. _P0-6._
- [ ] **H2 · Build + freeze golden template** (WORK-01) — run `provision-template.sh` in a CT, then
  `pct template <vmid>`; confirm Ubuntu 24.04 + Node 22 + pinned `claude-code@2.1.170` + ttyd + enabled
  `burrow-worker.service`. _P0-1 / STAR-GOLDEN-TEMPLATE._
- [ ] **H3 · Least-priv token + scoped ACL** (SETUP-02/03) — the `burrow@pve!burrow` privsep token
  resolves to pool/template/storage/node rights only (no out-of-pool VMIDs); the 9-priv role is
  enforced; a scoped `--full` clone succeeds. _P0-4._

**Boot + terminal**
- [ ] **H4 · ttyd LAN reachability** (WORK-04, SC-9) — from the control plane, `curl http://<worker-ip>:7681/`
  answers (bound to `0.0.0.0`, not `lo`). _P0-2._
- [ ] **H5 · Persistent ttyd survives detach** (SC-8, UI-05) — open ttyd, launch `claude`, close the
  tab, reconnect → the session is still alive (no `--once`). _P0-3 / P2-3._
- [ ] **H6 · Real clone + start + static net0** (WS-02) — create against a live node: `--full` clone from
  the template, clone/start UPIDs block to OK, `net0` carries the VMID-derived static IP. _P1-HV1._
- [ ] **H7 · Real ttyd health → running** (WS-02, WORK-04) — saga step 6 (`_wait_ttyd`) succeeds against a
  real worker on :7681; workspace → `running`. _P1-HV2._
- [ ] **H8 · Compensation tears down a real partial clone** (WS-03) — force a mid-saga failure → the
  partial CT is destroyed, the VMID freed, the row lands `error`, no orphan remains. _P1-HV3._
- [ ] **H9 · ⭐ Five-step acceptance gate** (SETUP-04, WS-01..08) — PRIMING.md STEP 4 end-to-end from a LAN
  browser: **create → live terminal → stop → start → destroy**; no orphan CTs, VMID pool clean. _P0-5 /
  P1-HV4. The "looks done but isn't" gate._
- [ ] **H10 · Live Claude TUI over real ttyd** (TERM-01/02/05) — the Claude Code TUI renders through the
  real `tty` subprotocol handshake and **reflows to the panel size** (not stuck 80×24). _P2-1._
- [ ] **H11 · Reattach after page refresh** (UI-05, TERM-06) — refresh the tab on a live session → reconnect

**Reproducible workers (Phase 3)**
- [ ] **H12 · Real worker boot → live Claude; boot.error on failure** (WORK-02) — re-run
  `20-create-template.sh`, boot a worker: fetches bootconfig → clones config+project → copies CLAUDE.md →

- [ ] **H14 · `x-access-token` credential convention** (WORK-02) — the GIT_ASKPASS `x-access-token`

- [ ] **H15 · Config-repo reachable** (WORK-02) — `cc-worker-config` clones at boot with the

- [ ] **H16 · `enabledPlugins` shape + plugin loads** (WORK-05) — on `claude-code@2.1.170`, the

**Fleet hardening (Phase 4)**
