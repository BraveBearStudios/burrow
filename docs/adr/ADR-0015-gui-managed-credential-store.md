# ADR-0015: GUI-managed encrypted credential store with a local admin gate

## Status

Accepted (supersedes the no-secret-at-rest stance of ADR-0011 and ADR-0012 only;
their singleton-table and provider-seam decisions stand)

## Context

Two credentials drive the control plane: the Proxmox API token (clone/start/stop/
destroy workers) and a GitHub credential the worker boot uses to clone the private
config and project repos. Both are `.env`-only today. ADR-0011 and ADR-0012 each
record "no secret enters the database" and deliberately decline a token-at-rest ADR;
the GitHub credential (`git_credential_token`, `config.py`) is likewise `.env`-only,
and when it is unset `mint_repo_credential` serves a `DEV-PLACEHOLDER` value, so a
private-repo clone fails at boot.

The operator needs to supply the GitHub PAT, and wants both credentials captured at
onboarding, stored securely, and create/update-able from the GUI after first-run
setup. "Update from the GUI" is unsatisfiable under `.env`-only for three independent
reasons: `settings = Settings()` is a frozen import-time singleton with no reload
hook; the api container does not (and should not) write the host's `0600`
`/opt/burrow/.env`; and Docker `env_file` is read only at container start. Delivering
in-GUI update therefore requires a secret-at-rest store, which reverses the
`.env`-only stance and is exactly the revisit trigger ADR-0012 named ("a future need
to persist or rotate the token").

The work must also meet SOC 2 Type 2 controls. Burrow is a self-hosted, LAN-only,
AGPL tool that runs on the operator's own infrastructure. The Brave Bear baseline
stack satisfies SOC 2 with Azure Key Vault (secrets), Entra External ID JWT (access
control), and Supabase (database), none of which exist in a self-host deployment.
Per the baseline's own deviation clause, the SOC 2 controls are applied as the
self-host model allows and every deviation is documented in this ADR rather than
silently dropped.

## Decision

Adopt a **write-only, encrypted-at-rest credential store** in the singleton
`settings` table, behind the `DbProvider` seam, gated by a **local admin secret**,
with **append-only audit logging**. The store holds the Proxmox API token and the
GitHub credential; a GUI write supersedes the `.env`/onboarding bootstrap value
live, without an api restart.

- **Encrypted at rest (Fernet).** Migration `004` adds nullable ciphertext +
  metadata columns to `settings` (`proxmox_token_enc`, `git_token_enc`,
  `proxmox_token_last4`, `git_token_last4`, `admin_secret_hash`,
  `credentials_updated_at`) plus an append-only `audit_log` table. Encryption is
  Fernet (AES-128-CBC + HMAC-SHA256) from the `cryptography` library; plaintext
  exists only transiently in memory at the encrypt step and at the proxmoxer / git
  boundary on read.

- **Pluggable key provider.** The Fernet key comes from a `SecretKeyProvider`
  abstraction. `EnvSecretKeyProvider` reads `BURROW_SECRET_KEY` (auto-generated by
  onboarding, `.env`-only, never in the DB) for the self-host path; a
  `KmsSecretKeyProvider` seam is reserved for the hosted path so Key Vault / KMS is
  additive, not a rewrite. If the store is enabled and no key resolves, the app
  fails fast at startup rather than running with an unusable store.

- **Write-only, never round-tripped.** The store value is never returned to any
  client. `GET /api/v1/setup/credentials` returns status only
  (`{ proxmoxTokenSet, proxmoxTokenLast4, gitTokenSet, gitTokenLast4, updatedAt }`).
  This is the same never-round-tripped guarantee SETUP-07 already locks, extended to
  the at-rest store.

- **Local admin gate (the SOC 2 access control).** A single operator admin secret is
  set during first-run setup (unauthenticated because nothing is yet protected),
  stored as an `argon2id` hash in `settings.admin_secret_hash`, never in `.env`.
  After setup, a `require_admin` dependency verifies an admin-secret request header
  against that hash and gates exactly the credential status + write endpoints and the
  post-setup settings screen. The rest of v1 stays LAN-only no-auth by design; only
  the most sensitive surface (the one holding the powerful PVE token and the PAT) is
  access-controlled.

- **Read precedence and no-restart update.** A credential present in the store
  overrides the `.env` value. On a Proxmox-token write the compute provider singleton
  is re-instantiated in-process (its proxmoxer client caches the token at `__init__`),
  so the new token takes effect with no container restart; the GitHub credential is
  read fresh from the store on each bootconfig mint, so it needs no rebuild. A new
  token is validated via the existing ephemeral throwaway-client `testConnection`
  path (ADR-0012) before it is persisted, so a broken token is never stored.

- **Audit logging (durable trail).** Every credential create / update / rotate and
  every admin-gate decision writes an append-only `audit_log` row and a structured
  JSON log line: action, source IP, which credential, timestamp. The secret value
  is never written to either. The `audit_log` table is never soft-deleted.

- **Scope.** Both credentials (Proxmox token + GitHub PAT) are in scope for onboarding
  capture and GUI create/update.

## SOC 2 Type 2 mapping

| Control | How this decision satisfies it |
|---|---|
| CC6.1 (secrets not in code) | Fernet key and credentials never in source; `.env.example` carries empty placeholders only; `cryptography` pinned. |
| CC6.7 (encryption at rest) | Credentials stored Fernet-encrypted; plaintext only transient in memory. |
| CC6.3 / CC6.6 (access control) | `require_admin` (argon2id) gates the credential surface and settings screen. |
| CC7.1 (error handling) | Fixed token-free envelope codes; no driver-exception or stack-trace leakage. |
| CC7.2 / CC7.3 (audit logging) | Append-only `audit_log` + structured JSON on every credential and admin-gate event; never the value. |
| CC6.8 (security headers) | Existing security-headers middleware verified on the new routes. |
| CC8.1 (change management) | Versioned migration `004` through the schema ledger; this ADR. |

## Deviations from the Brave Bear baseline (documented per the deviation clause)

| Baseline control | Self-host deviation | Justification |
|---|---|---|
| Secrets in Azure Key Vault | Fernet key in `.env` via `EnvSecretKeyProvider`; KMS provider seam reserved | No Azure subscription in a homelab self-host; the key-provider seam keeps Key Vault additive for the hosted path. |
| Entra JWT on every route | Local argon2id admin gate on the credential surface; rest of v1 LAN-only no-auth | No Entra tenant reachable from a self-host install; the network is the control boundary for non-credential routes (same posture as ttyd, ADR-0007). |
| Supabase / Postgres | SQLite `settings` table behind `DbProvider` | v1 self-host is SQLite-first (ADR-0001); the seam keeps a Postgres/hosted path additive. |

## Consequences

**Positive:**

- The operator can set and rotate both credentials from the GUI with no restart, and
  capture them at onboarding; the worker-boot clone gets a real credential.
- The most sensitive surface gains real access control and a durable audit trail,
  raising the self-host security posture above the prior `.env`-only no-auth state.
- The key-provider, `DbProvider`, and compute-provider seams keep the hosted path
  (Key Vault, Entra, Postgres) additive rather than a rewrite.

**Negative / trade-offs:**

- A secret-at-rest surface now exists; its safety rests on the Fernet key staying out
  of the DB and the leak test that locks plaintext out of every cell, envelope, and
  log. Key loss means the stored credentials cannot be decrypted and must be re-entered.
- A new `cryptography` dependency and an `argon2` dependency enter the api.
- The admin gate is a single shared operator secret, not per-user identity; it is the
  self-host stopgap until the hosted path brings Entra. This is the documented limit
  of access control under no-auth.

**Neutral / follow-on:**

- The leak test is extended to permit exactly the encrypted cells (ciphertext is not
  the plaintext sentinel) while still asserting no plaintext leak anywhere else.
- `tech-spec.md`, `REQUIREMENTS.md`, and `ROADMAP.md` rows that record the secret as
  out-of-scope-at-rest are updated to point at this ADR.

## Operational note: key loss recovery

Onboarding auto-generates `BURROW_SECRET_KEY` into the gitignored `.env`
(`40-control-plane.sh`). Losing it (a wiped `.env`, a backup restored without it)
makes the stored ciphertext unrecoverable: re-enter both credentials on the GUI
Credentials screen to re-encrypt under a fresh key. A missing or rotated key never
crashes the control plane or worker boot: `CredentialResolver` and the `main.py`
startup lifespan fall back to the `.env` values and log the mismatch loudly.

## Revisit trigger

A hosted multi-tenant path where credential access must be per-user (which brings
Entra JWT and retires the single-secret admin gate), or a requirement for envelope
encryption / external KMS as the primary key store (which activates the
`KmsSecretKeyProvider` seam and may retire the `.env` key for that deployment).
