## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator browser → wizard form fields | the Proxmox token is typed into a transient form field (step 1), sent to the Phase 12 redacted /setup/test-connection |
| React state → query cache / storage | the token must NOT cross into any persisted store (Zustand/localStorage/query cache) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-04 | Information disclosure | the token argument to useTestConnection | mitigate | The token is only a transient argument to `useTestConnection().mutate(body)`; useSetup.ts NEVER writes it to the query cache, Zustand, or localStorage. The mutation result type is ConnectionResult (no token echoed back). The gate state ["setupState"] holds only a timestamp. |
| T-13-05 | Information disclosure | client-side logging | mitigate | No console.log/console.error of the token or the test-connection body in useSetup.ts or NewWorkspaceModal. Server-side redaction is Phase 12 (SETUP-07); the client just must not log it. |
| T-13-06 | Tampering | persistent flag submission | accept | persistent is a non-sensitive boolean; the backend WorkspaceCreate already validates/accepts it (Phase 10). No new trust boundary. |
| T-13-SC | Tampering | npm installs | mitigate | No new npm packages added — uses only existing deps (@tanstack/react-query, react, MSW, vitest). No legitimacy checkpoint required. |

No unmitigated HIGH. The token is transient form state only (no client-side store, no client-side log).
