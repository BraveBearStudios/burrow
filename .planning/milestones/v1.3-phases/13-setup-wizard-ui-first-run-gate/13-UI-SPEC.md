# Phase 13 — UI Design Contract

> Visual and interaction contract for `SetupWizard.tsx` (full-page first-run gate), the four
> auto-advancing setup steps, and the `persistent` checkbox added to `NewWorkspaceModal`.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **This phase LOCKS to the existing Burrow design system — it introduces NO new colors,
> fonts, spacing, or radii.** Every value below is an existing CSS custom property declared in
> `ui/src/index.css:32-89` (four themes: `dark` / `dark-soft` / `medium` / `light`). The wizard
> reuses the `NewWorkspaceModal.tsx` overlay + container + `<Field>` + button idioms verbatim.

## Design System

| Property | Value |

## Color

Burrow is a dark-first four-theme system; every value is a `var(--token)` resolved per
`[data-theme]`. The 60/30/10 split below is expressed in the **default `dark` theme** (the hero
theme) but holds proportionally across all four because each theme block redeclares the full set.

| Role | Token (dark value) | Usage |
|------|--------------------|-------|
| Dominant (~60%) | `--bg` `#1a1c1a` | The full-page gate scrim backdrop fill (app behind the wizard) |
| Secondary (~30%) | `--bg-surf` `#212321` (card) + `--bg-panel-alt` `#2e302e` (inputs/secondary btn) | Wizard card surface, input fields, Cancel/secondary button, missing-priv list background |
| Accent (~10%) | `--accent` `#344734` (+ `--accent-line` `#5e7d5e` focus ring) | See reserved-for list below |
| Destructive / error | `--err` `#e05050` | Failing-step glyph `✕`, error strip left-border + text, invalid-field input border |

**Accent (`--accent` / `--accent-line`) reserved for — explicit list (never "all interactive"):**

1. The primary forward CTA on each step (`Validate`, `Verify`, `Re-check`, `Create workspace`) — green fill `var(--accent)` + `var(--btn-pri-text)` text, identical to the modal `Create` button.
2. The completed-step glyph `✓` and the completed-step row text (`--text`, with the `✓` mark; accent wash is NOT used per-row to avoid green sprawl).
3. The global `:focus-visible` ring on any wizard control (`--accent-line`, already app-wide in index.css:295).

**Other semantic tokens (status, NOT counted in the 10% accent budget — they are state signals):**

| Token | Use |
|-------|-----|
| `--ok` `#4ade80` | Health-check "db ok" / "compute ok" status dot + the passed-health row |
| `--warn` `#fbbf24` | (available for an in-progress/degraded health readout if needed) |
| `--text-muted` `#546654` | Pending-step glyph `○` and pending-step row text; inactive copy |
| `--gold` `#f0a737` | **RESERVED — do NOT reuse.** Gold is the `StepSpinner` top-arc ONLY (active-step spinner) + StatusBar stats. Never a fill, never a CTA, never a generic accent. |

Color discipline carried from the shipped contract: status is **never color-only** — every status
glyph/dot is paired with a text label (a11y, criterion 7). The missing-privilege list and all error
states pair the `--err` color with literal text.

## Copywriting Contract

The wizard is a 4-step vertical flow; copy below is the binding source of truth for the executor.
Backend error codes are FIXED and token-free (`api/main.py` `_SAFE_ERROR_MESSAGES`) — the UI surfaces
those messages plus the inline guidance below.

### Wizard gate + step structure

| Element | Copy |
|---------|------|
| Gate title (card header) | `Set up Burrow` |
| Gate subtitle (under title, `--text-sub`) | `Connect your Proxmox host and create your first workspace.` |
| Step indicator (4 labels) | `1 Connection` · `2 Template` · `3 Health` · `4 First workspace` |

### Step 1 — Token validation (`POST /setup/test-connection`)

| Element | Copy |
|---------|------|
| Step title | `Validate Proxmox connection` |
| Fields (reuse `<Field>`) | `Host` (placeholder `https://pve.lan:8006`) · `User` (placeholder `burrow@pve`) · `Token name` (placeholder `burrow-token`) · `Token value` (password input, placeholder `••••••••`) |
| Token helper (always visible, `--text-sub`) | `Validated in memory only — never stored. Keep the real token in your .env.` |
| Primary CTA | `Validate connection` |
| Success (auto-advance) | row marks `✓` `Connection OK` then advances to step 2 |
| Missing-privilege result (`success=false`, 200 — NOT an error) | Heading `Token is valid but missing privileges:` + a mono bulleted list of `missingPrivileges[]`; body line `Grant these on the Burrow privsep token, then re-validate.`; CTA stays `Validate connection` |
| Error `setup_unreachable` (502) | `Could not reach the Proxmox host.` + guidance `Check the host URL and that the API port (8006) is reachable, then retry.` |
| Error `setup_auth_failed` (400) | `The Proxmox token was rejected.` + guidance `Re-check the user, token name, and token value, then retry.` |

### Step 2 — Template verify (`POST /setup/verify-template`)

| Element | Copy |
|---------|------|
| Step title | `Verify worker template` |
| Fields | `Template VMID` (number, placeholder `9000`) · `Node` (placeholder `pve`) |
| Primary CTA | `Verify template` |
| Success (`usable=true`, auto-advance) | `✓ Template <vmid> on <node> is ready` then advance to step 3 |
| Not usable (`exists=true, usable=false`, 200) | `VMID <vmid> exists on <node> but is not a template.` + `Convert it to a template (or check the VMID), then re-verify.` |
| Error `setup_template_not_found` / `exists=false` | `The worker template was not found on the target node.` + `Confirm the template VMID and node, then re-verify.` |

### Step 3 — Health (`GET /api/v1/health`, degrade-not-500)

| Element | Copy |
|---------|------|
| Step title | `Check control-plane health` |
| Readout (two dotted rows) | `Database` → `--ok` dot + `ok` / `--err` dot + `unreachable` · `Compute` → `--ok` dot + `ok` / `--err` dot + `unreachable` |
| Primary CTA | `Re-check` (re-runs the health probe) |
| Success (both ok, auto-advance) | advances to step 4 |
| Degraded | `Burrow can start, but cannot reach Proxmox yet.` + `Fix the connection above, then Re-check.` |

### Step 4 — Create first workspace (reuse the create-form fields)

| Element | Copy |
|---------|------|
| Step title | `Create your first workspace` |
| Fields | reuse `NewWorkspaceModal` form: `Name`, `Git repo`, `Branch` (default `main`), `Node` (`Auto (least-loaded)`), `Persistent` checkbox |
| Primary CTA | `Create workspace` |
| On success | `POST /setup/complete` → gate re-renders away to the workspace list (criterion 2 ordering: complete AFTER create succeeds) |
| Create error (envelope, e.g. `capacity_exceeded`) | surface `error.message` verbatim in an `--err` strip; CTA stays `Create workspace` (retry inline) |

### Cross-cutting copy

| Element | Copy |
|---------|------|
| Empty / initial state heading | n/a — the gate is never "empty"; an unconfigured Burrow opens directly on step 1. The re-entry probe lands on the first failing step (criterion 3). |
| Empty state body | While re-probing on open: a single mono line `Checking setup…` with the `StepSpinner`, replaced by the first failing step. |
| Generic step error fallback | `Something went wrong. Try again.` (only if an un-coded error escapes) |
| Persistent checkbox label (`NewWorkspaceModal`) | `Persistent (keep workspace after session ends)` — verbatim from CONTEXT.md; default UNCHECKED (= ephemeral) |
| Destructive confirmation | **none** — this phase has zero destructive actions. The wizard only validates (read-only) and creates; setup-complete is idempotent; no delete/destroy/overwrite path exists. |

**Destructive actions in this phase:** none. (No confirmation pattern required.)

## Interaction & Layout Contract

> This section is the prescriptive layout/interaction contract for the executor. It is NOT in the
> base template but is required for the checker's Visuals dimension and the planner's task breakdown.

### The first-run gate (`App.tsx`)

- `App.tsx` calls `useSetupState()` (TanStack Query, `queryKey ["setupState"]`, refetch on mount). While `isLoading`, render nothing but the themed `--bg` root (no flash of the workspace list).
- When `setupState.setupCompletedAt === null` → render `<SetupWizard />` as the ONLY content inside the themed root, replacing the Navbar + middle row + StatusBar. When set → render the normal shell unchanged. The wizard never co-renders with the workspace list (hard gate).
- On the step-4 create success → `useCompleteSetup` mutation → invalidate `["setupState"]` → the gate re-renders away to the workspace list. No manual navigation.

### Wizard card (full-page modal)

- Same overlay idiom as `NewWorkspaceModal`: a fixed full-viewport scrim `background: rgba(8,13,8,.55)` over `--bg`, `display: grid; place-items: center`, `z-index: 50`. **Difference from the create modal: the gate scrim has NO click-to-dismiss and NO `✕` close** — it is a blocking gate, not a dismissible dialog (the only exits are completing setup or fixing a step).
- Card: `width: var(--w-modal)` (400px), `max-width: calc(100vw - 32px)`, `background: var(--bg-surf)`, `border: 0.5px solid var(--border-mid)`, `border-radius: var(--radius-card)` (12px), column flex — identical to `modalStyle`.
- `role="dialog"`, `aria-modal="true"`, `aria-label="Set up Burrow"`, `tabIndex={-1}`, focused on mount (matches the modal a11y idiom).

### Step indicator (locked to the existing checklist idiom — default, derived from `NewWorkspaceModal`)

- A **vertical 4-row checklist** rendered at the top of the card body (mono, 12.5px), reusing the exact glyph contract already shipped in `NewWorkspaceModal` saga: pending `○` (`--text-muted`), active = the `StepSpinner` ring (gold top-arc — its ONE sanctioned use), passed `✓` (`--text`), failed `✕` (`--err`). Each row = glyph + `9px` gap + label (`1 Connection`, `2 Template`, `3 Health`, `4 First workspace`).
- Why this and not a new horizontal stepper: the project already owns and ships this exact vertical-checklist-with-status-glyph pattern; reusing it keeps one idiom and zero new visual vocabulary. (This is the single design question the orchestrator flagged; locked to the established idiom rather than introducing a stepper.)
- Below the checklist: the ACTIVE step's form/readout + its primary CTA. Only the active step shows inputs; passed steps collapse to their `✓` row, pending steps to their `○` row.

### Per-step state matrix (every step declares all four)

| State | Visual |
|-------|--------|
| Idle / editing | Active-step `<Field>`s editable; primary CTA enabled when required fields are non-empty (mirrors `isValid` gating); checklist row shows `○`→active marker |
| Loading (in-flight mutation) | Active-step row shows `StepSpinner`; primary CTA disabled + `opacity: 0.5` + `cursor: not-allowed` (verbatim modal disabled style); inputs remain visible |
| Success | Row marks `✓`; **auto-advance** to the next step (CONTEXT.md decision); on the final step, gate closes |
| Error / partial | Row marks `✕` (`--err`); an `--err` left-bordered strip (`borderLeft: 2px solid var(--err)`, `background: var(--bg-panel-alt)`) shows the fixed message + inline guidance; inputs stay populated for inline retry (no persisted checkpoint) |

### Missing-privilege list presentation (step 1)

- Rendered as a mono (`--font-mono`, 12.5px) bulleted list inside a `--bg-panel-alt` block with a `--border` hairline and `--radius-control` (8px) — NOT an `--err` strip (a valid-but-under-privileged token is the SUCCESS path, `success=false`/200, per `setup.py`). One privilege name per line. Heading copy + guidance per the copywriting table.

### Health-check display (step 3)

- Two rows, each a status-dot + label pair (reuse the `StatusBar` `CountChip` dot idiom: `--sz-status-dot` 7px, `--radius-full`): `Database` and `Compute`, each `--ok` "ok" or `--err` "unreachable". Status is never color-only (the literal `ok`/`unreachable` text accompanies every dot).

### Button hierarchy (verbatim from the modal)

- **Primary CTA:** `background: var(--accent)`, `color: var(--btn-pri-text)`, no border, `height: 32px`, `padding: 0 14px`, `border-radius: var(--radius-control)`, `13px / 500`. One per step, right-aligned in a footer.
- **No secondary "Back"/"Cancel" on the gate** by default: it is a forward-only blocking gate with inline retry (CONTEXT.md: no persisted checkpoint, re-probe on open). If the implementer adds a "Back" affordance it MUST be the secondary style (`var(--bg-panel-alt)` + `0.5px solid var(--border-mid)`), never accent.

### Focus / keyboard / a11y for a blocking gate

- Card focused on mount; focus trapped within the card (the only interactive surface). `:focus-visible` paints the global `--accent-line` ring (index.css:295).
- `Enter` submits the active step's primary CTA when its required fields are valid (mirrors the modal `onKeyDown` Enter-submit). **`Escape` does NOTHING** — the gate is non-dismissible (override the modal's Esc-closes behavior).
- Each step's status region is an `aria-live="polite"` `role="status"` block (reuse the modal saga region) so success/error transitions are announced.
- Honor `prefers-reduced-motion` (already global in index.css:265) — the `StepSpinner` stills.

### Persistent checkbox (`NewWorkspaceModal`, WSX-02 UI half)

- A native `<input type="checkbox">` + label placed AFTER the Node selector row (after `NewWorkspaceModal.tsx:412-433`), in the form column (gap `14px`). Default UNCHECKED.
- Visual: checkbox `accent-color: var(--accent)` (so the checked fill is the green accent, not browser blue), `8px` gap to the label; label text `13px / 400 / --text`, copy `Persistent (keep workspace after session ends)`.
- Wires `const [persistent, setPersistent] = useState(false)` into the `createWorkspace.mutateAsync({ …, persistent })` body. The backend already accepts `persistent` (Phase 10) — no backend change.
