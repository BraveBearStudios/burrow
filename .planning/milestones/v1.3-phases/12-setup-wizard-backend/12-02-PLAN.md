<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| wizard client → /api/v1/setup/* | The operator-typed PVE token (HIGH-value asset) crosses the API boundary as a transient SecretStr request-body field and is handed to the compute seam for in-memory validation only. |
| router → compute seam | The router reads the raw token via `.get_secret_value()` ONLY at the `compute.testConnection(...)` call; it never logs the body, never echoes the token in an error envelope. |
| app → logs / DB / response envelope | The token must never reach a log line, a DB row, or a data/error envelope - proven RED-if-regressed by the sentinel-leak hard gate. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-04 (token-in-response) | Information Disclosure | the data/error envelope | mitigate | Endpoints return only ConnectionResult/TemplateResult (no token field); setup errors emit FIXED token-free messages from `_SAFE_ERROR_MESSAGES` - never `str(exc)`. The sentinel-leak test asserts the token is absent from both `data` and `error`. |
| T-12-05 (token-at-rest, ingress) | Information Disclosure | the SQLite store | mitigate | The setup endpoints write nothing to the DB (`getSetupState` is read-only; the setter is Phase 13). The sentinel-leak test scans EVERY table (driven off `sqlite_master`) and asserts the token is in no row - RED if any future write path persists it. |
| T-12-06 (token-in-logs, request path) | Information Disclosure | structured JSON logs + driver loggers | mitigate | The router never logs the body; the JsonFormatter `_ALLOWED_EXTRA_KEYS` whitelist drops a token-shaped extra; Plan 01's logger suppression pins the driver loggers to WARNING. The sentinel-leak test captures log output and asserts the token is absent. |
| T-12-07 (over-privileged / resource-creating validation) | Elevation of Privilege | the validation path | mitigate | The read-only/zero-resource test drives the REAL provider over the mocked tier and asserts `responses.calls` are GET-only (no POST/PUT/DELETE), proving SETUP-01 "creates zero resources." |
| T-12-SC | Tampering | npm/pip/cargo installs | accept | This plan adds NO new package-manager installs (only first-party routers + tests + an ADR; `responses` is already a dev dependency used by the Phase 10 mocked tier). No install task → no supply-chain checkpoint needed. |

No unmitigated HIGH threat remains: the token never enters an envelope, a DB row, or a log line, and the
validation path is proven GET-only / zero-resource. The sentinel-leak hard gate (T-12-04/05/06) is the
RED-if-regressed proof; the read-only assertion (T-12-07) is the zero-resource proof.
