<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| wizard client → setup validation | The operator-typed PVE token (HIGH-value asset: full clone/start/stop/destroy authority over the homelab) enters as transient request-body input and is handed to the compute seam for in-memory validation only. |
| compute seam → Proxmox node | The ephemeral validation client issues exactly one read-only `GET /access/permissions` (and template GETs) over CA-pinned TLS; it creates nothing. |
| app → logs / DB | Structured JSON logs + the SQLite store; neither must ever receive the token. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-01 (token-at-rest) | Information Disclosure | `proxmox_token_value` config + the validation token value | mitigate | Both are `SecretStr`; the real value is read ONLY at the proxmoxer boundary via `.get_secret_value()`. `testConnection` validates the request-body token in-memory via a throwaway client and NEVER writes it to the DB (`getSetupState` is read-only; no setter this phase) or to any field. |
| T-12-02 (token-in-logs) | Information Disclosure | proxmoxer/urllib3/requests driver loggers + exception strings | mitigate | `setup_logging()` pins those three loggers to WARNING so the driver cannot echo the Authorization header / token at DEBUG; `SecretStr.__repr__` masks the value in any 422 / validation-error / `f"{settings}"`. Setup errors raise FIXED token-free `SetupAuthError`/`SetupUnreachableError` messages - no raw proxmoxer exception string is interpolated. |
| T-12-03 (over-privileged-validation) | Elevation of Privilege | the ephemeral validation client | mitigate | The client issues ONLY read-only GETs (`/access/permissions`, template config) over CA-pinned TLS (`verify_ssl=ca_cert_path`, never disabled); it builds zero resources, never assigns to `self._api`, and asserts capability against the documented 9-priv set rather than attempting any mutation. |
| T-12-SC | Tampering | npm/pip/cargo installs | accept | This plan adds NO new package-manager installs (SecretStr is already in the pydantic dependency; no new deps). No install task → no supply-chain checkpoint needed. |

No unmitigated HIGH threat remains: the token is SecretStr-wrapped, validated in-memory by a read-only
ephemeral client, kept out of logs by logger suppression + token-free errors, and never written to the
DB (the sentinel-leak hard-gate proof lands in Plan 02 over this foundation).
</threat_model>
