<objective>
Extend the frozen Phase-0 SQLite seam so it can serve as the race-safe VMID-reservation arbiter and event-log source the Phase-1 saga depends on, and land every Phase-1 `Settings` key in one place. This is the foundation wave: every other backend plan reserves VMIDs through this DB, reads events through it, and reads config from these settings.

Purpose: SC-3/SC-4 (WS-10) make the DB partial unique index — not the in-process `getNextVmid` scan — the authority that prevents two live workspaces sharing a VMID, while surviving destroy-then-recreate. WS-11 needs an event-log read method. The saga's compensation lookups need workspace-by-vmid. Consolidating all new `Settings` keys here keeps `config.py` a single-owner file so no two later plans collide on it.
Output: `002_vmid_unique.sql`, an ordered `migrate()` that actually applies it, `VmidTakenError` raised on duplicate-active-vmid INSERT, `getEvents` + `getByVmid` on the ABC and SqliteProvider, all Phase-1 `Settings` keys, and an integration test proving race-safety and VMID reuse over real SQLite.
</objective>
