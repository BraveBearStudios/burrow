  add `persistent` to both, defaulting via `data.get("persistent", False)` (mirrors the

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> /api/v1/workspaces | `persistent` is operator input through the create body; validated server-side by Pydantic `WorkspaceCreate` (CamelModel). |
| service -> SQLite | The `003` DDL + the `persistent` value cross into storage; both must be literal/parameterized. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-03A | Tampering | SQL injection via migration / column value | mitigate | `003` is a static `.sql` literal (no interpolation, fixed `DEFAULT 0`); `persistent` flows through the parameterized INSERT (`:persistent` bind), never string-formatted (V5; verified all `sqliteProvider` writes are bound params). |
| T-10-03B | Tampering | a non-bool `persistent` body crashing the API | mitigate | Pydantic `WorkspaceCreate.persistent: bool` rejects a non-bool body at the boundary -> 422 envelope, not a raw 500 (V5 input validation). |
| T-10-03C | Information Disclosure | `settings.setupCompletedAt` mistaken for a secret store | mitigate | ADR-0011 records it as a NON-sensitive timestamp only; the Proxmox token stays `.env`-only (token-at-rest ADR avoided by design — Phase 12 concern). No secret is added by this phase. |
| T-10-03D | Denial of Service | a second `settings` row silently creating a divergent config | mitigate | `CHECK (id = 1)` + a single seeded row; the migration test asserts a second insert collides. |
| T-10-SC | Tampering | npm/pip/cargo installs | accept | No package installs — every lib (`aiosqlite`, `pytest`) is already committed; RESEARCH Package Legitimacy Audit is N/A. |
</threat_model>
