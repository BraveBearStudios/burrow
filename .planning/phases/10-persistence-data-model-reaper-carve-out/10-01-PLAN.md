(respx patches httpx; proxmoxer rides requests — RESEARCH Pitfall 3/5).

proxmoxer 2.3.0 verified shapes (from api/.venv site-packages, cited in RESEARCH):
- `Tasks.blocking_status` (proxmoxer/tools/tasks.py:31-41): decodes the node from the
  UPID, then polls `prox.nodes(node).tasks(task_id).status.get()` until
  `data["status"] == "stopped"`. proxmoxer unwraps the `data` envelope, so the registered
  status GET body is `{"data": {"status": ..., "exitstatus": ...}}`.
- `decode_upid` (proxmoxer/tools/tasks.py:53-55): asserts the UPID is exactly 9
  colon-separated segments starting `UPID:`, else raises
  `AssertionError("UPID is not in the correct format")` (RESEARCH Pitfall 4).
- `ResourceException` constructor (proxmoxer/core.py:57):
  `ResourceException(status_code, status_message, content, errors=None, exit_code=None)`.

Real provider entry points the self-tests must drive (NOT the Fake):
- `ProxmoxComputeProvider.startCt/stopCt/destroyCt` (api/compute/proxmoxProvider.py:241-282)
  -> `_block(upid)` UPID polling (proxmoxProvider.py:67-96).
- Defensive inspectors `_is_not_found` / `_is_running_or_locked`
  (proxmoxProvider.py:337-367) key on `status_code` + message text.

Base URL + UPID shape (test_proxmox_provider.py:32-44, 282-283):
  `_BASE = f"https://{host}:8006/api2/json"`
  9-segment UPID: `UPID:{node}:0000ABCD:00100000:64000000:{ttype}:{vmid}:burrow@pve:`
  (segments: UPID, node, pid, pstart, starttime, type, id, user, comment[empty ok]).
