Add the operator-facing "Auto (least-loaded)" path to the create modal so an
operator can decline a manual node pick (WSX-01 "operator does not pick a node"):

   pick still works, and the form is valid with Auto.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator -> create modal | the operator chooses Auto or a manual node; the form gates submission |
| modal -> POST /api/v1/workspaces | the create payload's `node` (null for Auto, a string for manual) crosses to the backend |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-03 | Tampering (input validation) | `NewWorkspaceModal` submit payload | mitigate | The modal sends either `node: null` (Auto) or a node string the operator picked from the `useNodes` list (which the backend derives from `worker_nodes`). It does not let an operator type an arbitrary node string into the create payload; the authoritative validation remains server-side (the step-0 guard + `selectNode` over `worker_nodes`). The UI is a convenience layer, not the trust boundary. |
| T-09-01 | Denial of Service (overcommit) | UI capacity display | accept | The modal only displays/selects nodes; the no-overcommit guarantee is enforced server-side (Plan 02 `selectNode` + step-0 guard). A stale UI capacity view cannot cause overcommit because the server re-checks at create time. |
| T-09-SC | Tampering (supply chain) | npm installs | accept | No new npm packages — the change uses the existing React/vitest/MSW harness (a type tweak, a `<select>` option, three tests). RESEARCH Package Legitimacy Gate not triggered. |
</threat_model>
