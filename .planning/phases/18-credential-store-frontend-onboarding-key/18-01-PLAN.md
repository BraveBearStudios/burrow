# Phase 18 Plan 01: Credential Backend Completion + Onboarding Key

**Requirements:** CRED-05 (backend), CRED-06, CRED-07
**Surface:** `api/` + `cc-worker-config/lxc/host-prime/40-control-plane.sh` + `.env.example`

## Task 1 — Audit read endpoint (CRED-05 backend)
- `DbProvider.listAudit(self, limit: int = 100) -> list[dict[str, Any]]` (ABC
  `@abstractmethod` in `api/db/provider.py`).
- SQLite impl (`api/db/sqliteProvider.py`): `SELECT id, action, target, outcome,
  sourceIp, detail, createdAt FROM audit_log ORDER BY createdAt DESC, rowid DESC
  LIMIT ?`; return list of dicts (camelCase keys already match columns).
- Postgres impl (`api/db/postgresProvider.py`): `NotImplementedError` stub for ABC
  parity (mirrors the other credential methods).
- `GET /setup/audit` in `api/routers/setup.py`, `dependencies=[Depends(require_admin)]`,
  optional `limit` query (clamp 1..500, default 100), returns `respond({"entries": [...]})`.

## Task 2 — BURROW_SECRET_KEY onboarding auto-gen + recovery doc (CRED-06)
- In `40-control-plane.sh`, after `.env` is assembled from `.env.example`: if the
  `BURROW_SECRET_KEY=` value is empty, generate a Fernet key
  (`python3 -c 'from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())'`)
  and write it into `.env` in place (umask 077, same gitignore-guard as the existing
  secret handling; never echo the key). Idempotent: a non-empty key is left untouched.
- Add a key-loss recovery note (comment in the script + a short doc line): losing the
  key makes stored ciphertext unrecoverable; re-enter credentials via the GUI. The
  resolver + startup already fall back to `.env`, so a missing/rotated key never
  crashes the control plane or worker boot.
- Verify the fallback: confirm `CredentialResolver` + `main.py` lifespan already
  swallow a missing/undecryptable key (they do — grep-confirmed); add a focused
  regression test only if a real gap is found.

## Task 3 — Extended sentinel leak test (CRED-07)
- Extend the setup credential-leak integration test: drive `POST /setup/credentials`
  with a sentinel token (admin-gated), then assert the sentinel plaintext appears in:
  NO `settings` cell and NO `audit_log` cell (query every row/column), NO API envelope
  (the save response, `GET /setup/credentials`, `GET /setup/audit`), and NO captured
  log line. Assert the POSITIVE: the `proxmoxTokenEnc`/`gitTokenEnc` BLOB is present
  and decrypts back to the sentinel, and the last4 == sentinel[-4:]. Prove RED-if-regressed
  by a transient "store the plaintext" mutation, then revert.
