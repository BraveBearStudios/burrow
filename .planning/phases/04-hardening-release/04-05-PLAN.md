<objective>
Author the release supply-chain workflow (CICD-05).

This plan creates `.github/workflows/release.yml` triggered on a `v*` tag / published release that builds and pushes both images to GHCR by digest, generates a syft SBOM (SPDX + CycloneDX), signs each image with cosign keyless (Sigstore + GitHub OIDC, no stored key), and emits a SLSA build-provenance attestation bound to the digest — with exactly the least-privilege per-job permissions and every third-party action SHA-pinned. GHCR is the project's own registry, so this is CI-verifiable (the actual push + cosign-verify against a published digest is the CD/human authority).

Purpose: Every published image is scanned, SBOM'd, signed, and provenance-attested by immutable digest, with no long-lived signing key and no over-broad token — the supply-chain integrity gate for the v1 release.
Output: `release.yml`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI publish job → GHCR | The job holds a token that can push images + attestations |
| OIDC provider → cosign/provenance | Keyless signing depends on the ephemeral OIDC identity |
| third-party GitHub Action → CI runner | A compromised/typosquatted action runs with the publish token |
| floating tag vs digest | Signing/attesting a tag instead of a digest leaves the binding spoofable |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05A | Spoofing / Information Disclosure | long-lived signing key | mitigate | cosign KEYLESS (Sigstore + GitHub OIDC) — no stored key to leak/rotate; ephemeral Fulcio cert recorded in Rekor (ci-cd §5.4 / State-of-the-Art) |
| T-04-05B | Elevation of Privilege | publish job token scope | mitigate | Per-job `permissions` are EXACTLY `contents: read, packages: write, id-token: write, attestations: write` — nothing broader; the workflow default stays `contents: read` (Pitfall 7 / ci-cd §5.5) |
| T-04-05C | Tampering | signing/attesting a floating tag | mitigate | Sign + attest the immutable `steps.build.outputs.digest`, never a tag — the signature/provenance binds the exact image (anti-pattern in RESEARCH) |
| T-04-05D | Tampering / Elevation | compromised third-party action | mitigate | Every action (checkout, login, metadata, setup-buildx, build-push, sbom-action, cosign-installer, attest-build-provenance) SHA-pinned with a version comment (Pitfall 14 / ci-cd §5.5) |
| T-04-05E | Spoofing | unsigned/unattested image reaching consumers | mitigate | Every published image is signed + dual-format SBOM'd + provenance-attested by digest; the verify invocation is documented for downstream `cosign verify` / `gh attestation verify` |
| T-04-05F | Information Disclosure | fork PR reading publish credentials | mitigate | Publishing is gated to `v*` tags / published releases (not PRs); fork PRs never trigger the publish job (ci-cd §5.5) |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | no new runtime dependency; the supply-chain tooling is SHA-pinned GitHub Actions in YAML — governed by the pin convention, nothing to slopcheck |
</threat_model>
